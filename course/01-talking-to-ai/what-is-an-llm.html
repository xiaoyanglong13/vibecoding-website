What is an LLM?

Before we talk about vibecoding, we should talk about what makes vibecoding possible. A class of AI models called large language models has crossed a threshold where you can ask for software in plain English and get something that works. This page is about what those models are, what they can do, and why their existence rewrites the rules of who gets to build.

The shift from how to what

For sixty years, working with a computer has been a one-way street. You describe how to do something — every step, every condition, in a language the machine understands — and the machine runs it. If you can't write the steps, you can't get the result. That barrier has kept "building software" in the hands of people who write code.

LLMs change the rules. You describe what you want — the outcome, the constraint, the desired shape of the answer — and the model produces a result. The steps are still happening underneath; you just stop being the one who has to write them.

This sounds incremental. The consequence runs deeper. The barrier to building things with a computer has shifted from syntax (do you know how to write the steps?) to articulation (can you describe what you want?). For most knowledge workers, articulation is a skill they already have.

A working definition

A large language model is a system trained on enormous amounts of text — books, websites, code, papers — with a deceptively simple objective: predict the next word in a sequence. Repeat that prediction billions of times, on a model with billions of parameters, and something surprising happens. The model develops abilities that were never explicitly trained.

A useful mental picture: an LLM is a very capable generalist who has read a substantial fraction of the public internet and held it all in working memory. The generalist isn't a database — it doesn't look things up — but it has internalized patterns from everything it's seen, and it can apply those patterns to your specific question.

Four capabilities that matter

You'll lean on four capabilities of modern LLMs constantly. They're worth knowing by name.

1 · Broad knowledge

Acquired through pre-training on a large fraction of public text. An LLM can hold a passable conversation about Renaissance painting, derivatives pricing, and the rules of cricket — not because anyone programmed it to, but because all of that material was in its training data. This is why you don't have to teach an LLM what "discounted cash flow" means; you can use the term and move on.

2 · Conversational interaction

Trained models can hold a multi-turn conversation, track context across turns, and respond to follow-up questions in plain language. This is the capability that makes LLMs usable by anyone who can type a sentence — no command line, no syntax, no training course required.

3 · In-context learning

The most underrated capability on this list. You can paste an example into a prompt — a short memo, a CSV row, a code snippet — and the model will infer the pattern and apply it to new material without any retraining. This is why showing the model an example of the format you want works so well: the model is learning from the example in real time. We'll lean on this heavily on the next page.

4 · Reasoning

Modern LLMs can break a problem into steps and work through them sequentially. The technique called chain-of-thought prompting — explicitly asking the model to show its work — reliably improves accuracy on problems that involve more than one inference step. This is the capability that makes LLMs useful for analysis, not just retrieval.

Two new kinds of work LLMs unlock

These four capabilities power two distinct things you can now do — each one genuinely new under the sun. Both run through this course.

Thinking with AI

Cognitive work that no software could do before. Reading open-ended customer feedback and extracting the insight. Drafting a fluent first version of a board memo. Structuring an ambiguous strategy question. Synthesizing fifty pages of user interviews into a coherent narrative. Brainstorming pricing options with judgment, not just a list.

Humans have written summaries and brainstormed ideas forever. What's categorically new is that something other than a person can now do this work — fast, on demand, and at scale. No traditional algorithm can read qualitative text and produce real insight. A script that tried would either count keywords (and miss the point) or wrap a call to an LLM (in which case it's doing the same cognitive work, just programmatically).

You've probably already done some thinking with AI inside a chatbot — asking ChatGPT to summarize an article, brainstorm gift ideas, or rewrite an awkward email. That's the family. Module 1 is largely about getting good at this kind of work, since the same skills carry over to the next family.

Vibecoding

Using AI to build software. The model writes code from your description; you describe what you want and what "done" looks like; the AI handles the syntax. This is where the leverage compounds — a working tool, once built, can be re-run forever. From Module 2 onward we focus on this family.

Software has existed forever. What's categorically new is who can produce it. A working tool used to require an engineer; now it requires articulation. The barrier moved.

The term vibecoding was coined by AI researcher Andrej Karpathy in February 2025:

"There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists. … It's not really coding — I just see stuff, say stuff, run stuff, and copy paste stuff, and it mostly works."

— Andrej Karpathy, @karpathy on X, February 2, 2025

The exact phrase matters less than what it points at: building working software without writing the code yourself.

How the two compose

Most real work uses both. You might think with AI to synthesize three vendor proposals into a shortlist (text in, insight out), then vibecode a small spreadsheet that scores them against criteria you can adjust (description in, working tool out). The model underneath is the same; the kind of work it's doing for you is different.

One pointer forward: in Module 2 you'll meet agents — LLMs that can read your files, run code, and act on the world rather than only respond to your messages. Agents extend both families. They let you do thinking with AI over your own documents instead of just what fits in a chat window, and they make vibecoding practical for real projects rather than toy scripts. For now, hold both families in mind as chat-based; we'll lift that constraint in Module 2.

Why this is revolutionary

Three things to keep in mind as the rest of the course unfolds.

The interface changed. The barrier to working with a computer used to be language — you had to learn the machine's. Now the machine speaks yours. Calling this a usability upgrade understates the change; it shifts the relationship into a different category.

The cost of expert work collapsed. Cognitive work that used to require a smart analyst and several hours — a qualitative synthesis, a first-cut memo, a structured breakdown of a fuzzy question — now takes twenty minutes and a careful prompt. A piece of software that used to require an engineer and two days of work costs about the same. Things you would never have done because they weren't worth the time become doable.

The target audience changed. Doing analyst-grade synthesis used to require an analyst. Building software used to require an engineer. Both are becoming general activities — the way using a spreadsheet did in the 1980s, or making a slide deck did in the 1990s. The people who learn to work with LLMs while the rest of their function is still catching up will compound an advantage that lasts.

Common confusion at this stage

  • "Isn't this just ChatGPT?" ChatGPT is one product built on top of an LLM. The capabilities described here are properties of the model itself, not of any particular interface. Claude, Gemini, and other modern models have similar capabilities.
  • "Do I need to understand how LLMs work under the hood?" No. You need a working mental model of what they can and can't do. This page is enough for that — the math is irrelevant to the skill.
  • "Why does the model sometimes sound confidently wrong?" Because it's predicting plausible text, and plausible isn't the same as correct. Module 5 takes this seriously.

Try this thought experiment

Before moving on, take 60 seconds and think of one task at work or school you'd ideally automate but assumed required engineering help. Hold onto it. By the end of Module 4 you'll know whether it's a good vibecoding candidate, and by the end of the course you'll probably have built it.
