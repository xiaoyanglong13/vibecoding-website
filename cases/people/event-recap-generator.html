Event recap generator

You just got back from a case competition. The club president needs a polished debrief by Friday — in the same format as always. Your raw material is one person's messy notes. Here's how to do it in one session.

Difficulty · Beginner ~20 minutes Uses: Modules 1, 4

1. The business decision

After every case competition, the club president sends a standard debrief — same sections, same tone, same length, every time. The problem: the person who takes notes is never the person who writes the recap, and turning messy bullet points into the finished document takes two to three hours. This workflow cuts that to one prompt. There's no code involved — the folder structure and CLAUDE.md do all the work.

2. Data available

Three files, each in its own folder:

recap-project/
  CLAUDE.md
  inputs/
    notes.txt
  outputs/
  references/
    competition-recap-fall-2024.docx

inputs/notes.txt — raw notes from whoever attended. Expect partial sentences, names spelled inconsistently, and sections that are either too thin or too detailed. A typical excerpt:

arrived 8:45, registration slow
case: retail supply chain disruption - 20 min prep
our team: maya, darius, priya + me
judges were tough on financials - darius handled well
placed 2nd overall, 1st on presentation score
debrief w/ judge afterwards - said our market sizing was off
but framework was strong
dinner sponsored by alumni - talked to james chen (mckinsey)

references/competition-recap-fall-2024.docx — the last approved recap. This is the format the AI needs to match. Every section, every heading, the tone. Download the sample reference file to follow along.

CLAUDE.md — the rules the reference file alone can't communicate. More on this below.

Why each folder exists

inputs/ holds raw, messy source material that will change every time you run this workflow. references/ holds stable examples the AI uses to understand what "correct" looks like — you'd update it rarely, only when the format itself changes. outputs/ is where finished work lands, kept separate so it's never confused with source material. CLAUDE.md captures institutional knowledge that isn't visible in the files themselves: who the audiences are, what's mandatory, what tone to use. Without it, the AI has to guess.

3. The build

Three rounds, about 20 minutes total.

Round 1 · Write the CLAUDE.md together

Don't write the CLAUDE.md alone. Instead, ask the AI to interview you:

I'm setting up a folder for generating recurring event recap
documents. I have a reference example and raw notes as inputs.

Before we write anything, ask me the questions you'd need
answered to write a good CLAUDE.md — the file that tells you
the rules for this project every time you open it.

Ask one question at a time.

The AI will surface things you hadn't thought to specify: whether certain sections are mandatory even if nothing happened in that area, how to handle names, whether to flag anything for human review before sending. Once you've answered its questions, ask it to draft the CLAUDE.md from your answers and review it before saving.

A well-written CLAUDE.md for this project looks something like:

# Event Recap Project

## Purpose
Generate a polished event recap document from raw notes,
matching the format in references/.

## Mandatory sections
The following sections must always appear, even if the notes
contain little or nothing for them:
- Event overview (date, location, format, team)
- Results
- Key learnings (minimum two)
- What we'd do differently
- Thank-yous (always include sponsors by name)

## Format rules
- Match the heading structure and approximate length of the
  reference document exactly
- Do not invent facts. If the notes are silent on something,
  write "[to confirm]" rather than guessing
- Names: use first name only for students, full name + affiliation
  for external contacts

## Output
Save the finished document to outputs/ as
  recap-[event-name]-[YYYY-MM].docx

Round 2 · Check understanding before generating

Before asking the AI to produce anything, ask it to tell you what it learned from the reference file. This catches misreads early:

Read the reference document in references/ and tell me:
1. What sections does it contain, in order?
2. What's the approximate length of each section?
3. What tone does it use — formal, casual, somewhere in between?
4. Anything in the format that seems like a deliberate choice
   rather than a default?

Don't generate the recap yet. Just describe what you see.

If its description doesn't match what you know the format to be, correct it here — before the document gets written.

Round 3 · Generate and review

With the CLAUDE.md in place and the format confirmed:

Now generate the recap. Use:
- inputs/notes.txt as the source material
- references/competition-recap-fall-2024.docx as the format
- CLAUDE.md for all rules about audience, mandatory sections,
  and tone

Where the notes are thin, write [to confirm] rather than
filling in details. Save the output to outputs/.

Read the output and check two things: every mandatory section is present, and any [to confirm] flags are things you can actually fill in. If a mandatory section is missing, the most common cause is a CLAUDE.md rule that was too vague — tighten it and re-run.

4. Key prompts

The prompts used across the three rounds, in order:

-- Round 1: build the CLAUDE.md --

I'm setting up a folder for generating recurring event recap
documents. I have a reference example and raw notes as inputs.
Before we write anything, ask me the questions you'd need
answered to write a good CLAUDE.md. Ask one question at a time.

-- Round 2: check understanding --

Read the reference document in references/ and tell me what
sections it contains, their approximate length, and the tone
it uses. Don't generate the recap yet.

-- Round 3: generate --

Now generate the recap using inputs/notes.txt as source,
references/ as the format guide, and CLAUDE.md for all rules.
Where notes are thin, write [to confirm]. Save to outputs/.

5. What good looks like

The most instructive thing you can do is run this workflow twice: once with a minimal CLAUDE.md and once with the full version above, using the same notes. The difference makes the case for why the instructions file matters more than the prompt.

With a thin CLAUDE.md With a detailed CLAUDE.md
Sections appear in a different order than the reference Section order matches the reference exactly
"Thank-yous" section is omitted because the notes don't mention it "Thank-yous" appears with a [to confirm] flag, reminding you to add names
Invented a plausible-sounding judge quote that wasn't in the notes Flagged the gap with [to confirm] instead of guessing

A good output lands in outputs/, matches the reference on structure and length, and contains no invented facts — only [to confirm] placeholders where the notes ran dry.

Watch for

  • No reference = invented structure. If you skip the references/ folder and just describe the format in words, the AI will produce something plausible but wrong. The reference file is not optional — it's the ground truth.
  • Missing mandatory sections. If a section isn't in the notes, the AI will silently omit it unless CLAUDE.md explicitly says it's required. Always list non-negotiable sections by name.
  • "It sounds right" isn't the same as "it matches." Read the output against the reference side by side before signing off. The AI is very good at writing polished prose that misses structural details you haven't noticed yet.

6. Extensions

Once the base workflow is running, try these to go further:

  • Run the workflow for three events in a row with different notes files. The folder stays the same — only inputs/ changes. This is the core payoff of the folder structure: the setup cost is paid once.
  • Add a rule to CLAUDE.md asking the AI to flag any "lessons learned" items that appear across multiple events, once you have three or more recaps in references/. The folder becomes institutional memory.

The multi-audience version maps to Module 5's section on output formatting; the cross-event pattern analysis is a good lead-in to Module 7.
