Model automation

You own the monthly three-statement update. Right now it eats half a day of copy-paste from the trial balance into a tired Excel model. Here's how to get it to fifteen minutes without giving up the model your CFO trusts.

Difficulty · Beginner ~45 minutes Uses: Modules 1, 4

1. The business decision

You're a first-year FP&A analyst at a mid-market manufacturer. On the second business day of every month, you update the three-statement model: income statement, balance sheet, cash flow. The CFO uses it on the third business day for the monthly business review.

Today, your process is: export the trial balance from NetSuite, open last month's workbook, paste the new numbers into the right rows, fix anything that broke, and reconcile until the balance sheet balances. It takes four to five hours and you've made sign-flip mistakes twice this quarter.

You don't need a new model. You need the same model, refreshed in fifteen minutes, with a short note explaining what changed.

2. Data available

Three things are sitting in a folder on your laptop. Sample versions of all three are linked below so you can follow along:

  • trial_balance.csv — April's TB exported from the GL. About 100 rows, single period. Each new month you re-export it.
  • account_mapping.csv — maps each GL account number to a line on the model (e.g. "Revenue", "COGS", "Accrued expenses") and a sign multiplier. You maintain this file by hand as the chart of accounts grows.
  • model.xlsx — the live three-statement model. The "Inputs" tab holds each period's numbers, one column per month. January through March are frozen historicals from prior monthly updates; the April column is laid out and empty, ready for the script to fill. The IS, BS, and CF tabs reference Inputs through named ranges, so populating April lights up the whole model.

The "add one fresh column to Inputs each month" step is the one bit of manual setup that survives the automation. It takes two minutes; the script does the rest.

The real-world messiness: a few accounts each month are new (e.g. a freshly opened legal entity), so the mapping file is always slightly behind. Sign convention is mixed — revenues are negative in the GL, positive in the model. And "Inputs" has six merged-cell sections that openpyxl will complain about.

A sample of the trial balance:

account_number, account_name, period, debit, credit
40000, Product revenue - US, 2026-04, 0, 1842317.55
40010, Product revenue - EU, 2026-04, 0, 612090.10
50000, COGS - materials, 2026-04, 734221.00, 0
60100, Salaries & wages, 2026-04, 488740.25, 0
21000, Accrued expenses, 2026-04, 0, 145620.00

3. The build

Total session was about an hour, in four rounds. The slowest part wasn't the code — it was getting the mapping and sign conventions right.

Round 1 · Plan before execute

I started with a plan request, not a code request. I gave it the role, the three files, and the exact steps I needed — including constraints that matter in practice:

You are a financial analyst. Every month you need to update
the three-statement model based on this month's trial balance
spreadsheet. You want to write Python code that automates the
paste-and-reconcile step. There are three relevant files in
the folder:
- trial_balance.csv
- account_mapping.csv
- model.xlsx

The steps should be:
1) For each row in the TB file, categorize its model_line
   using the account_mapping. If no mapping exists, skip but
   record in a separate file.
2) For each row in the last column of the Input worksheet of
   model.xlsx, sum up the corresponding debit-credit for that
   model_line from the TB file. Fill the cell.
3) All existing files are read-only. Create new files if needed.
4) Save the updated model to model_2026-04.xlsx, do not
   overwrite source.
5) Leave all formula cells intact, do not change.

Before writing any code, first show me the complete plan.

The plan it returned was solid: it laid out each step, called out the four accounts that would land in the unmapped file, and explained the sign convention it would apply. I approved it and moved on.

Round 2 · Execute

Once the plan was approved, I gave the go-ahead and added the four checks I wanted printed at the end — in the same message:

Go ahead and write the code. At the end I want to print:
- Any account_numbers in the TB that weren't in the mapping
- Total revenue, total opex, net income for the period
- Whether the BS balances (assets - liabilities - equity,
  should be ~0)
- Whether IS net income matches the change in retained
  earnings on the BS

It produced about 60 lines of Python using pandas and openpyxl. First run failed because the Inputs tab had merged cells in the header row. I pasted the error back and it switched to writing by named range instead of cell coordinates, which was cleaner anyway.

Round 3 · Challenge the summary

The script printed five unmapped accounts and a balance sheet off by $1,240. The model pointed to the new lease liability as the culprit for the BS gap — which was right — but glossed over the other four accounts because they happened to net to zero: the LATAM revenue pilot (+$1,000) and an "Other income" line (+$500) were exactly offset by two new EU expense accounts (−$800 and −$700).

I pushed back: even if net income doesn't move, individual IS lines are still misstated — revenue is understated by $1,500 and SG&A is understated by $1,500. Coincidental netting isn't the same as correct. I asked it to add all five accounts to the mapping. After that re-run: zero unmapped accounts, BS residual exactly $0.00, and a $167K retained-earnings mismatch that remained — correctly, because that's a real open accounting question (likely a dividend or equity adjustment) that no mapping change can fix.

Round 4 · Summary table

With the model balancing cleanly, I asked the script to also print a formatted summary table for the CFO note:

I want my code to generate a summary table that lists:
1. Revenue (value for this month, percentage change from
   last month)
2. Net income (value for this month, percentage change from
   last month)
3. Balance sheet (whether it is balanced, how many new
   accounts added)

It read the prior-period column from the source model to compute the month-over-month changes and printed the table in markdown. That output is what you see in section 5 below — the script produces it automatically every run.

4. Working code

The final script, lightly cleaned. You'd save this as update_model.py next to the three input files:

import pandas as pd
from openpyxl import load_workbook

PERIOD = "2026-04"

tb = pd.read_csv("trial_balance.csv")
mp = pd.read_csv("account_mapping.csv")

# --- Net amount per account, then apply mapping sign ---
tb["net"] = tb["credit"] - tb["debit"]
merged = tb.merge(mp, on="account_number", how="left")

unmapped = merged[merged["model_line"].isna()]
if len(unmapped):
    print("WARNING - unmapped accounts:")
    for _, r in unmapped.iterrows():
        print(f"  {int(r['account_number'])}  {r['account_name']}")

merged["signed"] = merged["net"] * merged["sign"].fillna(0)
by_line = merged.groupby("model_line")["signed"].sum()

# --- Write into model.xlsx via defined names ---
# Each model line has a defined name like "in_revenue_2026_04".
wb = load_workbook("model.xlsx")
for line, value in by_line.items():
    slug = (line.lower()
            .replace("&", "and")
            .replace(" ", "_")
            .replace("-", "_"))
    while "__" in slug:
        slug = slug.replace("__", "_")
    name = f"in_{slug.strip('_')}_{PERIOD.replace('-', '_')}"
    if name in wb.defined_names:
        dest = wb.defined_names[name].value
        sheet, cell = dest.split("!")
        wb[sheet.strip("'")][cell.replace("$", "")] = float(value)

wb.save(f"model_{PERIOD}.xlsx")

# --- Checks ---
def L(name): return by_line.get(name, 0)

revenue = L("Revenue")
opex    = L("COGS") + L("SG&A") + L("R&D") + L("Depreciation")
other   = L("Interest income") + L("Interest expense") + L("Tax expense")
ni      = revenue + opex + other

print(f"Revenue:    {revenue:,.0f}")
print(f"Opex:       {opex:,.0f}")
print(f"Net income: {ni:,.0f}")

assets = sum(L(x) for x in
             ["Cash", "AR", "Inventory", "Prepaid expenses",
              "PPE", "Intangibles"])
liab   = sum(L(x) for x in
             ["AP", "Accrued expenses",
              "Lease liability ST", "Lease liability LT", "Debt"])
equity = L("Common stock") + L("Retained earnings prior") + ni
print(f"BS check (A - L - E): {assets - liab - equity:,.2f}")

Model line names in the mapping file use spaces only (no dashes or other punctuation) so the slug calculation produces names that match the workbook's defined names exactly. The script's WARNING block surfaces any TB account that didn't find a mapping — that's how the lease-liability mismatch in Round 3 showed up.

5. What good looks like

The deliverable isn't the script — it's the model file plus a three-line note to the CFO. The script prints this table automatically at the end of every run; you paste it into the note and add a sentence of context. The note is the part most analysts skip, and it's the part that builds trust:

ItemThis monthvs. last monthComment
Revenue $2.77M +1.3% EU and APAC held steady; small LATAM pilot launched.
Net income $288K -10% Materials cost ran hot; new EU entity overhead just kicked in.
Balance sheet Balanced — New short-term lease-liability account added to mapping.

Once you trust the script, the fifteen minutes is: drop the new TB in the folder, run the script, eyeball the three checks, write the note, send.

Watch for

  • New accounts that look like old ones. An account named "Lease liability - ST" looks harmless until you realize it's never been on the balance sheet before. Always read the unmapped-accounts warning before saving.
  • Sign conventions. GLs and models disagree. Get the sign column in the mapping right once, then trust it — but spot-check revenue and net income each month against last month's printed model.
  • Letting the script overwrite the source model. Always write to a dated file. If the script has a bug, you want last month's working model still on disk.
  • Skipping the human note. A script that produces a balanced model with no commentary is less useful than a half-day spreadsheet with a sentence of insight. Don't lose the last mile.

6. Folder structure for a routine workflow

There's a subtle friction in the setup above: the script reads model.xlsx as its source, but it writes model_2026-04.xlsx as its output. That means every month you'd need to rename last month's output back to model.xlsx before running again — which is exactly the kind of manual step that causes mistakes and defeats the purpose of automating the task.

The fix is a clean folder structure and one small change to the script. Move the three input files into an inputs/ subfolder, and point all script output to an outputs/ subfolder:

monthly-update/
  inputs/
    trial_balance.csv      ← replace this each month
    account_mapping.csv    ← maintain as chart of accounts grows
    model.xlsx             ← the original template; never touched again
  outputs/
    model_2026-01.xlsx
    model_2026-02.xlsx
    model_2026-03.xlsx
    model_2026-04.xlsx     ← this month's run
  update_model.py

Then update the script to find its source automatically — the most recent file in outputs/, falling back to the template on the very first run:

import glob, os

existing = sorted(glob.glob("outputs/model_????-??.xlsx"))
source = existing[-1] if existing else "inputs/model.xlsx"
wb = load_workbook(source)

# ... rest of the script unchanged ...

wb.save(f"outputs/model_{PERIOD}.xlsx")

Now the monthly routine is exactly three steps: drop the new trial balance into inputs/, update PERIOD at the top of the script, run it. The outputs/ folder also gives you a free audit trail — every prior month's model is still on disk if you need to go back.

Watch for

  • A corrupted output becoming next month's source. The glob picks the most recent file alphabetically, so a bad run will propagate. Always eyeball the three checks before closing the terminal — if anything looks wrong, delete the output and fix the issue before the next run picks it up.

7. Extensions

Once the basic update is reliable, ask the agent to:

  • Add a variance-vs-budget column and flag any line item more than 5% off plan.
  • Auto-draft the three-line CFO note from the period-over-period deltas, for you to edit rather than write from scratch.
  • Wrap the script as a scheduled task that runs on the first business day, emails you the unmapped-accounts list, and waits for you to update the mapping before producing the final model.

The scheduled-task version maps to Module 7's section on hooks and recurring jobs.
